## 1. 项目概述

### 1.1 背景

在软件规模度量领域，COSMIC（Common Software Measurement International Consortium）功能规模度量方法（ISO/IEC 19761）是业界公认的标准。然而，将自然语言需求文档转换为符合 COSMIC 标准的功能用户需求（FUR）文档，以及进一步生成需求规格说明书（SRS，IEEE 830），需要大量专业人工工作。

本平台利用大模型 AI 技术，实现：
1. **需求文档 → COSMIC 文档** 的智能转换
2. **COSMIC 文档 → SRS 文档** 的自动建模

同时提供全流程的项目管理、人工修改追踪、管理员审核等协作功能。

### 1.2 目标用户

| 角色 | 描述 | 核心功能 |
|------|------|----------|
| **普通用户** | 项目经理、需求分析师、软件度量工程师 | 创建项目、上传文档、触发 AI 转换、在线编辑、追踪修改量 |
| **管理员/审核员** | 质量管理人员、技术负责人 | 审核 COSMIC/SRS 文档、批注意见、通过/驳回 |
| **系统** | 后台 AI 服务 | 自动执行文档转换、版本管理、日志记录 |

### 1.3 核心价值

- **效率提升**: AI 自动完成需求 → COSMIC → SRS 的转换，缩短 80% 人工工作量
- **质量可控**: 人工编辑追踪 + 管理员审核双保障
- **精准度量**: 精确的字符级修改比例统计，量化人工 vs AI 贡献
- **三文档绑定**: 需求文档、COSMIC 文档、SRS 文档在同一项目中统一管理

## 3. 系统架构

### 3.1 三文档体系

```
┌─────────────────┐      AI 转换       ┌─────────────────┐      AI 转换       ┌─────────────────┐
│   需求文档       │ ────────────────→ │  COSMIC 文档    │ ────────────────→ │   SRS 文档      │
│  (Word/.docx)   │                   │ (Excel/.xlsx)   │                   │  (Word/.docx)   │
│                 │                   │                 │                   │                 │
│ · 自然语言需求   │                   │ · 功能过程(FP)  │                   │ · 功能需求描述   │
│ · 业务场景描述   │                   │ · 触发事件(TE)  │                   │ · 接口定义      │
│ · 用户故事      │                   │ · 数据移动(DM)  │                   │ · 非功能需求    │
│                 │                   │ · CFP 度量值    │                   │ · 验收标准      │
└─────────────────┘                   └─────────────────┘                   └─────────────────┘
      ↑                                                    ↑                         ↑
      └──────────── 人工上传/编辑 ─────────────────────────┘                         │
            └──────────── 人工编辑/审核 ─────────────────────────────────────────────┘
```

### 3.2 文档流转状态

```
文(需求文档) → 表(COSMIC) → 规(SRS) → 审(审核完成)
   [已完成]      [AI生成中]     [待生成]      [未开始]
```

每个文档具有独立状态：
- **需求文档**: 已上传 / 已确认
- **COSMIC 文档**: AI生成中 / 待确认 / 编辑中 / 审核中 / 已通过 / 已驳回
- **SRS 文档**: 待生成 / 生成中 / 待审核 / 已通过 / 已驳回

### 3.3 多迭代版本管理（v1.1 新增）

一个项目支持多个独立迭代（如 v1.0、v2.0），每个迭代从零开始走完整流程：

```
项目 A
├── v1.0 (当前) — 需求 → COSMIC → SRS → 审核通过
│   └── 已发布
├── v1.1 (历史) — 需求 → COSMIC → SRS → 审核通过
│   └── 已归档
└── v2.0 (进行中) — 需求 → (COSMIC 编辑中)
    └── 仅含需求文档
```

**迭代特性**：
- 每个迭代独立拥有三文档，新迭代不继承旧文档内容
- 通过 `project_iterations` 表管理，项目页可通过下拉选择器切换当前迭代
- `previous_iteration_id` 记录溯源关系
- `change_summary` 描述本轮需求变更原因
- 项目整体状态由「当前迭代」状态驱动

**迭代状态**（与项目整体状态保持一致）：
- `draft` — 初始状态
- `converting` — AI 转换进行中
- `reviewing` — 等待审核
- `approved` — 审核通过

## 5. AI 转换流程

### 5.1 需求文档 → COSMIC 文档

```
用户上传 .docx 需求文档
        ↓
系统解析文档内容（文本提取、结构化）
        ↓
AI Agent 分析（多步骤，后台执行，不阻塞用户）:
  步骤 1: 识别功能用户（Functional Users）
  步骤 2: 识别功能过程（Functional Processes）
  步骤 3: 识别触发事件（Trigger Events）
  步骤 4: 分析数据移动（Data Movements: Entry, Exit, Read, Write）
  步骤 5: 计算 CFP 总值
        ↓
生成 COSMIC 文档（.xlsx 格式）
        ↓
通知用户完成，进入 "待确认/编辑中" 状态
```

### 5.2 COSMIC 文档 → SRS 文档

```
用户触发 "AI 转 SRS"（COSMIC 需已完成）
        ↓
AI Agent 基于 COSMIC FUR 建模:
  - 转换功能过程为功能需求描述
  - 推导接口需求
  - 根据数据移动推导数据需求
  - 补充标准 SRS 章节（引言、总体描述、详细需求等）
        ↓
生成 SRS 文档（.docx 格式，IEEE 830 标准）
        ↓
通知用户完成
```

### 5.3 后台执行策略

- AI 转换在后台异步执行，用户无需等待
- 工作台显示实时进度条（步骤 N/5）
- 转换期间文档状态为 "AI转换中"
- 支持取消转换

## 6. 编辑与修改追踪

### 6.1 追踪机制

| 层级 | 追踪内容 | 标识方式 |
|------|----------|----------|
| 字符级 | 新增/删除字符数 | 右下角标签显示 +/-N字 |
| 段落级 | 哪些段落被修改 | 左侧边框颜色 |
| 文档级 | AI vs 人工比例 | 统计面板百分比 |
| 会话级 | 本次编辑时长 | 计时器 |

### 6.2 版本管理

- 每次保存生成新版本号
- AI 生成初始版本为 v1.0
- 每次人工保存递增小版本 v1.1, v1.2...
- 提交审核时锁定版本
- 支持历史版本对比与回滚

### 6.3 编辑历史记录

- 记录每次编辑操作: 时间、操作人、操作类型、影响范围
- 操作类型: AI生成、人工修改、新增段落、删除段落、采纳AI建议

## 7. 审核工作流

### 7.1 审核状态流转

```
                    ┌──────────┐
        ┌──────────→│  编辑中   │←──────────┐
        │           └────┬─────┘            │
        │                │ 提交审核          │
        │                ↓                   │
   驳回 │           ┌──────────┐            │ 编辑修改
        │           │  待审核   │            │
        │           └────┬─────┘            │
        │                │                   │
        └────────┬───────┴───────┬──────────┘
                 ↓               ↓
            ┌──────────┐    ┌──────────┐
            │  已通过   │    │  已驳回   │
            └──────────┘    └──────────┘
```

### 7.2 审核权限

- 仅管理员/审核员角色可进行审核操作
- 审核时需填写审核意见
- 驳回时文档退回编辑中状态，用户按意见修改后重新提交

### 7.3 Diff 对比

审核时自动对比 AI 生成版本（v1.0）与当前提交版本（vN）：
- 展示所有新增/删除/修改内容
- 支持行级高亮
- 右侧显示质量评分辅助决策

## 8. 数据模型

数据模型与数据库表结构（详见 SDD）保持完全一致，使用 SQLite 存储。

### 8.1 项目（Project）

```
Project {
  id: UUID
  name: String               // 项目名称
  creator_id: UUID           // 创建者
  description: Text          // 项目描述
  context_json: Text         // 静态业务背景知识（JSON，AI Prompt 上下文）
  current_iteration_id: UUID // 当前活跃迭代 FK → project_iterations.id
  status: Enum               // 整体状态: draft/converting/reviewing/approved（自动同步自当前迭代）
  is_deleted: Boolean        // 0-未删除, 1-已删除
  created_at: DateTime
  updated_at: DateTime
  members: [Member]          // 项目成员列表（独立 project_members 表）
  iterations: [Iteration]    // 项目迭代列表
}

Member {
  user_id: UUID
  role: Enum                 // admin / editor / reviewer / viewer
  joined_at: DateTime
}

Iteration (ProjectIteration) {
  id: UUID
  name: String               // 迭代名称，如 v1.0、v2.0
  status: Enum               // draft / converting / reviewing / approved
  requirement_doc_id: UUID   // 需求文档 FK → documents.id
  cosmic_doc_id: UUID        // COSMIC 文档 FK → documents.id
  srs_doc_id: UUID           // SRS 文档 FK → documents.id
  previous_iteration_id: UUID// 上一迭代 FK（溯源）
  change_summary: Text       // 本轮变更摘要
  is_current: Boolean        // 是否当前活跃版本（项目主页默认展示）
  published_at: DateTime     // 审核完成发布时间
}
```

### 8.2 文档（Document）

```
Document {
  id: UUID
  project_id: UUID
  type: Enum             // requirement / cosmic / srs
  version: String        // v1.0, v1.1 etc.
  source: Enum           // ai / manual_upload / manual_edit
  status: Enum           // pending / generating / editing / reviewing / approved / rejected
  title: String
  original_filename: String  // 原始上传文件名
  file_path: String      // 本地存储相对路径（指向 uploads/ 目录）
  content_json: Text     // 结构化内容（JSON 字符串，用于编辑器渲染）
  diff_stats: {          // 修改统计
    total_chars: Int
    ai_chars: Int
    human_added: Int
    human_deleted: Int
    modified_blocks: Int
  }
  is_deleted: Boolean    // 0-未删除, 1-已删除
  created_at: DateTime
  updated_at: DateTime
}
```

### 8.3 编辑记录（EditRecord）

```
EditRecord {
  id: UUID
  document_id: UUID      // 关联文档
  action: Enum           // ai_generate / human_edit / accept_suggestion / create / delete
  target: String         // 目标段落/字段
  detail: String         // 详细描述
  delta: Int             // 字符变化量 (+N / -N)
  is_deleted: Boolean    // 0-未删除, 1-已删除
  created_by: UUID       // 操作人（系统操作为 null）
  created_at: DateTime
}
```

### 8.4 审核记录（AuditRecord）

```
AuditRecord {
  id: UUID
  document_id: UUID
  version: String        // 被审核版本
  submitter_id: UUID     // 提交人
  reviewer_id: UUID      // 审核人
  status: Enum            // pending / approved / rejected
  comment: Text           // 审核意见
  scores: {              // 质量评分
    completeness: Int     // 完整性
    accuracy: Int         // 准确性
    consistency: Int      // 一致性
    compliance: Int       // 规范性
  }
  is_deleted: Boolean    // 0-未删除, 1-已删除
  submitted_at: DateTime
  reviewed_at: DateTime
}
```

### 8.5 项目级 AI 迭代记忆（ProjectMemory）

```
ProjectMemory {
  id: UUID
  project_id: UUID       // 所属项目
  category: Enum          // correction(修正) / rule(规则) / preference(偏好) / example(案例)
  key: String             // 关键词标签，用于检索匹配（如"功能过程拆分"、"数据移动类型"）
  content: String         // 记忆的具体规则/偏好文本（AI 生成时遵循）
  source: Enum            // system(自动提炼) / human_edit(用户编辑触发) / audit(审核驳回触发) / manual(用户手动添加)
  hit_count: Int          // 被检索命中次数，用于排序和衰减淘汰
  last_used_at: DateTime  // 上次被检索使用的时间
  is_deleted: Boolean     // 0-未删除, 1-已删除
  created_at: DateTime
}
```

**说明**：
- `project_memories` 与 `projects.context_json` 的分层设计：
  - `context_json`（存于 projects 表）：项目静态背景知识（领域、功能用户、术语表），全量注入每个 Prompt
  - `project_memories`（本表）：动态迭代修正偏好，按需检索注入（按当前步骤关键词匹配），支持 hit_count 衰减淘汰
- 自动生成：用户每次人工编辑 COSMIC/SRS 文档并保存后，系统对比 AI 原始输出与修改后的差异，自动提炼记忆存入本表
- 手动维护：用户可在「项目设置 > AI 偏好」中手动添加/修改/删除硬规则
- 使用时机：AI 转换流水线的每个步骤前，根据步骤关键词检索最相关的 5-8 条记忆注入 Prompt
- 衰减机制：6 个月未使用且 hit_count <= 2 的记忆自动软删除；手动添加的 `rule` 类型永不过期
